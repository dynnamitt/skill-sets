# Accessibility & Component Design for Federated Search

Cherry-picked from [Building Components](https://github.com/anthropics/skills)
accessibility and composition guides.

## Compound component API

Distribute responsibility across cooperating subcomponents instead of one
monolithic `<Search>` with dozens of props:

```tsx
import * as Search from '@/components/FederatedSearch'

<Search.Root onSelect={handleSelect} onQueryChange={setQuery}>
  <Search.Input placeholder="Search products, categories..." />
  <Search.Dropdown>
    <Search.Group label="Products" icon={<ProductIcon />}>
      {products.map(p => (
        <Search.Item key={p.id} value={p}>
          {p.name}
        </Search.Item>
      ))}
    </Search.Group>
    <Search.Group label="Categories" icon={<CategoryIcon />}>
      {categories.map(c => (
        <Search.Item key={c.id} value={c}>
          {c.name}
        </Search.Item>
      ))}
    </Search.Group>
  </Search.Dropdown>
</Search.Root>
```

### Component responsibilities

| Component | Role |
|-----------|------|
| `Root` | Context provider: query state, active index, keyboard handler |
| `Input` | `role="combobox"`, `aria-expanded`, `aria-controls`, `aria-activedescendant` |
| `Dropdown` | `role="listbox"`, positioned overlay |
| `Group` | `role="group"` + `aria-label={label}`, visual header/separator |
| `Item` | `role="option"`, `aria-selected`, two action zones (append / navigate) |

## ARIA for combobox with grouped results

The ARIA combobox pattern needs careful wiring when results are grouped:

```tsx
// Input
<input
  role="combobox"
  aria-expanded={isOpen}
  aria-controls="search-listbox"
  aria-activedescendant={activeItemId}
  aria-autocomplete="list"
  aria-label="Search products, categories, and brands"
/>

// Dropdown
<div id="search-listbox" role="listbox" aria-label="Search results">
  {/* Each group */}
  <div role="group" aria-label="Products">
    <div role="presentation" className="group-header">Products</div>
    <div role="option" id="item-p1" aria-selected={activeId === 'item-p1'}>
      Product result 1
    </div>
  </div>
</div>
```

Key points:
- `aria-activedescendant` on the input tracks the focused option (not DOM focus)
- DOM focus stays on the input at all times — arrow keys update `aria-activedescendant`
- Each `role="group"` needs an `aria-label` so screen readers announce the category
- Group headers are `role="presentation"` — they're visual-only, not focusable

## Live region for streaming results

As async results arrive, announce counts to screen readers:

```tsx
<div aria-live="polite" aria-atomic="true" className="sr-only">
  {isLoading
    ? 'Searching...'
    : `${totalCount} results: ${productCount} products, ${categoryCount} categories`
  }
</div>
```

Use `aria-busy={true}` on the listbox while any source is still loading,
`aria-busy={false}` when all sources have resolved:

```tsx
<div
  role="listbox"
  aria-busy={sources.some(s => s.isLoading)}
  aria-label="Search results"
>
  {/* groups */}
</div>
```

## Keyboard navigation across groups

Arrow keys must traverse items across group boundaries seamlessly:

```tsx
function handleKeyDown(e: React.KeyboardEvent) {
  const allItems = flattenGroupItems(groups) // flat list of all items across groups

  switch (e.key) {
    case 'ArrowDown':
      e.preventDefault()
      setActiveIdx(prev => (prev < allItems.length - 1 ? prev + 1 : 0))
      break
    case 'ArrowUp':
      e.preventDefault()
      setActiveIdx(prev => (prev > 0 ? prev - 1 : allItems.length - 1))
      break
    case 'Enter':
      e.preventDefault()
      if (activeIdx >= 0) handleSelect(allItems[activeIdx])
      break
    case 'Escape':
      e.preventDefault()
      closeDropdown()
      break
    case 'Home':
      e.preventDefault()
      setActiveIdx(0)
      break
    case 'End':
      e.preventDefault()
      setActiveIdx(allItems.length - 1)
      break
  }
}
```

Items are indexed as a flat list — group boundaries are invisible to keyboard nav.
This keeps the interaction simple: arrow through everything, Enter to select.

## Two-action items

Each suggestion can either **append a filter prefix to the query** or
**navigate directly** to the result. Visually separate these with
distinct click targets or keyboard shortcuts:

```tsx
<div role="option" id={itemId} aria-selected={isActive}>
  <span className="item-label" onClick={() => navigateTo(item)}>
    {item.name}
  </span>
  <button
    tabIndex={-1}
    aria-label={`Add ${item.type}:${item.name} to search`}
    onClick={() => appendPrefix(`${item.type}:${item.name}`)}
  >
    <PlusIcon aria-hidden="true" />
  </button>
</div>
```

The primary action (Enter key) should navigate. The secondary action
(append prefix) is a supplementary affordance for power users who want
to refine their query incrementally.

## Focus management

- Focus moves to the input on mount (or when opened via Cmd+K)
- Focus stays on the input while navigating — `aria-activedescendant` tracks the visual highlight
- When the dropdown closes (Escape, blur, selection), restore focus to the input
- The dropdown should not trap focus — Tab should move to the next focusable element

## Virtualization

For groups with many results (>50 items), virtualize with `virtua` or
use `content-visibility: auto` to avoid rendering hundreds of DOM nodes.
Most federated search UIs show only 3-5 items per group, so this is
rarely needed — but keep it in mind if you show "see all N results" expanded views.
